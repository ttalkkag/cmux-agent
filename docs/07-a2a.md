# A2A 프로토콜 적용 검토

> A2A 프로토콜의 개념을 현재 cmux-agent에 어떻게 적용할지 검토한 문서.
> 이 프로젝트는 사내 개발자가 로컬에서 사용하는 내부 개발 툴이므로, 인증·공개 엔드포인트 등 외부 노출 관련 사항은 다루지 않는다.

---

## 1. 현행 구조와 A2A 매핑

### 1.1 실제 런타임 구조

현재 구현은 "파일 기반 durable channel"과 "cmux 터미널 주입 기반 UX channel"이 겹쳐 있는 구조다.

1. `cmux_agent/cli/commands.py`의 `cmd_start()`가 `Run`을 생성하고, `.cmux/agents.json`을 읽어 `controller`, `orchestrator`,
   `worker-*` agent를 등록한다.
2. `cmux_agent/cli/commands.py`의 `cmd_task()`는 첫 사용자 요청을 outbox가 아니라 orchestrator 터미널로 직접 주입한다. 즉, 작업의 "입구"부터 파일
   transport를 거치지는 않는다.
3. orchestrator/worker AI CLI는 `.cmux/outbox/*.json`에 `dispatch` 또는 `result` artifact를 직접 쓴다.
4. `cmux_agent/application/watcher.py`의 `ArtifactWatcher`와 `_OutboxHandler`가 파일 생성/이동 이벤트를 감지하고 `type`, `sender`,
   `recipient`, `message` 4개 필드를 검증한다.
5. `cmux_agent/application/broker.py`의 `MessageBroker`가 artifact를 `Message` 레코드로 저장한 뒤`AgentFileSystem.write_to_inbox()`
   로 수신자 inbox JSON을 만든다.
6. 같은 브로커가 `PromptBuilder.build_injection_prompt()`와 `CmuxAdapter.send_text()`를 사용해 recipient 터미널에 자연어 프롬프트를 다시 주입한다.

즉, A2A 도입 시 치환 대상은 2~4번의 durable control plane이고, 5번의 cmux 터미널 주입은 로컬 UX 어댑터로 남기는 편이 자연스럽다.

### 1.2 도메인/인프라/애플리케이션 매핑

| 현재 개념           | 실제 코드 위치                                                                                         | A2A에서 가장 가까운 개념                                                | 코드 기준 판단                                                                                                                                                                        |
|-----------------|--------------------------------------------------------------------------------------------------|----------------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `Run`           | `cmux_agent/domain/models.py`의 `Run`, `RunStatus`                                                | 세션/컨텍스트 래퍼에 가까움                                                | `Run`은 workspace 전체 생명주기(`CREATED/RUNNING/COMPLETED/FAILED`)만 추적한다. A2A `Task`처럼 개별 작업별 상태나 history/artifacts를 갖지 않는다. 굳이 A2A 쪽에 붙이면 `task`보다는 `contextId` 전구체에 더 가깝다.          |
| `Agent`         | `cmux_agent/domain/models.py`의 `Agent`, `cmux_agent/cli/commands.py`의 `cmd_start()`              | `AgentCard`의 로컬 런타임 엔트리                                        | 현재는 agent 이름, role, `surface_id`만 저장한다. A2A `AgentCard`가 요구하는 `supportedInterfaces`, `skills`, `capabilities`가 없다.                                                              |
| `Message`       | `cmux_agent/domain/models.py`의 `Message`, `cmux_agent/infrastructure/storage.py`의 `messages` 테이블 | A2A `Message`의 축약판                                             | `sender`, `recipient`, `type`, `payload`, `status`만 있고 `contextId`, `taskId`, `parts`, `metadata`, `referenceTaskIds`가 없다. 현재 상태(`PENDING/DELIVERED/FAILED`)도 작업 상태가 아니라 큐 상태다. |
| artifact 감지/라우팅 | `cmux_agent/application/watcher.py`, `cmux_agent/application/broker.py`                          | `SendMessage` 요청 처리                                            | 실제로는 JSON 파일을 감지해서 recipient inbox에 복제하는 로컬 브로커다. | 이벤트 로그          | `cmux_agent/domain/events.py`, `cmux_agent/infrastructure/event_log.py`                          | `TaskStatusUpdateEvent`, `TaskArtifactUpdateEvent`의 내부 감사로그 버전 | append-only JSONL은 내부 운영 기록으로는 충분하지만, A2A처럼 외부 client가 구독하는 표준 event stream은 아니다.                                                                                               |
| SQLite 상태 저장소   | `cmux_agent/infrastructure/storage.py`의 `StateStore`                                             | A2A 서버 내부 상태 저장                                                | 내부 저장소라는 점에서는 그대로 유지 가능하다. 다만 현재 스키마는 `runs`, `agents`, `messages`만 있어 A2A `Task`/`Artifact` 저장 구조가 없다.                                                                         |

### 1.3 `.cmux/agents.json`과 Agent Card 비교

현재 저장소의 `.cmux/agents.json`은 다음과 같은 "agent 이름 → 실행할 CLI 명령" 맵이다.

```json
{
    "orchestrator": "claude",
    "worker-1": "claude",
    "worker-2": "codex",
    "worker-3": "gemini"
}
```

`cmux_agent/cli/commands.py`의 `_load_config()`와 `cmd_start()`는 이 값을 provider 문자열로만 사용한다. invalid JSON이면 기본값으로 조용히
fallback하고, `worker-` 접두사가 붙은 key만 동적 worker로 취급한다. 따라서 이 파일은 Agent Card가 아니라 launcher config에 가깝다.

| A2A `AgentCard` 필드                        | 현재 상태 | 근거                                                          |
|-------------------------------------------|-------|-------------------------------------------------------------|
| `name`, `description`, `version`          | 없음    | `.cmux/agents.json`에는 provider 문자열만 있다.                     |
| `supportedInterfaces`                     | 없음    | HTTP/JSON-RPC/gRPC endpoint를 서빙하는 코드가 없다.                   |
| `capabilities`                            | 없음    | streaming, push notification, extended card 여부를 선언할 위치가 없다. |
| `defaultInputModes`, `defaultOutputModes` | 없음    | 현재 prompt/message는 text 전제지만 스키마로 선언하지 않는다.                 |
| `skills`                                  | 없음    | 역할별 prompt는 존재하지만 machine-readable skill 목록은 없다.            |

결론적으로 `.cmux/agents.json`을 그대로 Agent Card로 재해석하면 안 된다. Phase 1에서는 "Agent Card에 필요한 메타데이터를 담을 수 있는 내부 정규화 구조"로 확장하는 것이 맞다.

### 1.4 현재 artifact / delivery 스키마와 A2A `Task` / `Message` 비교

현재 watcher가 인정하는 canonical outbox artifact는 최소 스키마다.

```json
{
    "type": "dispatch",
    "sender": "orchestrator",
    "recipient": "worker-2",
    "message": "docs/a2a.md의 섹션 4~6을 보강하세요."
}
```

실제 처리된 artifact 예시(`.cmux/processed/dispatch-worker-2.json`, `.cmux/processed/result-worker-1.json`)도 동일하게 `type`,
`sender`, `recipient`, `message`만 사용한다. 반면 inbox delivery는 `PromptBuilder.build_delivery()`가 다음 구조로 만든다.

```python
return {
    "message_id": None,
    "from": sender,
    "type": msg_type.value.lower(),
    "message": payload.get("message", ""),
    "context": payload.get("context", {}),
    "created_at": now,
}
```

이 구현에는 중요한 제약이 두 가지 있다.

- SQLite `messages.message_id`와 inbox 파일명에는 UUID가 있지만, JSON body의 `message_id`는 항상 `null`이다.
- `taskId`, `contextId`, `parts`, `metadata`가 없어 A2A `Message`로 곧바로 승격할 수 없다.

매핑을 코드 기준으로 다시 쓰면 아래와 같다.

| 현재 필드/개념                       | 실제 코드 위치                                                                                                     | A2A 대응물                                             | Gap                                                             |
|--------------------------------|--------------------------------------------------------------------------------------------------------------|-----------------------------------------------------|-----------------------------------------------------------------|
| outbox `message`               | `cmux_agent/application/watcher.py`, `.cmux/processed/*.json`                                                | `Message.parts[0].text` 또는 `Artifact.parts[0].text` | 텍스트만 가능하고 media type/structured part가 없다.                       |
| outbox `sender`, `recipient`   | `cmux_agent/application/broker.py`                                                                           | 라우팅 메타데이터                                           | A2A `Message` 본문에는 recipient가 없고 endpoint/interface 수준에서 라우팅한다. |
| outbox `type=dispatch/result`  | `cmux_agent/application/watcher.py::validate_artifact()`                                                     | `SendMessage` 요청/응답 혹은 `Task.status` 전이             | 현재는 로컬 workflow 구분자일 뿐이며 A2A의 표준 필드는 아니다.                       |
| inbox `message_id: null`       | `cmux_agent/application/prompting.py::build_delivery()`                                                      | `Message.messageId`                                 | 식별자를 잃고 있어 A2A correlation에 직접 쓰기 어렵다.                          |
| inbox `context`                | `cmux_agent/application/prompting.py::build_delivery()`                                                      | `metadata` 또는 `data` part                           | 현재는 자유 dict이며 schema나 part kind가 없다.                            |
| `Message.payload`에 raw JSON 저장 | `cmux_agent/application/broker.py::_route_message()`, `cmux_agent/infrastructure/storage.py::save_message()` | `Task.history` + `Message` + `Artifact`             | 지금은 원본 payload 전체를 문자열로 저장할 뿐, task/artifact 단위로 분해하지 않는다.      |

A2A 공식 모델과의 가장 정확한 대응은 다음과 같다.

- 현재 `dispatch` artifact는 A2A `SendMessage`의 입력 `message`와 가장 가깝다.
- 현재 `result` artifact는 A2A `Task.status.message` 또는 최종 `Artifact.parts[0].text`와 가장 가깝다.
- 현재 코드에는 A2A `Task` 자체에 대응하는 영속 엔티티가 없다. 가장 가까운 것은 `Run` + `Message` 조합이지만, 이는 작업 단위가 아니라 세션 단위 기록이다.

### 1.5 Gap 분석 검증

기존 Gap 요약은 방향은 맞지만, 코드 기준으로는 아래처럼 더 구체적으로 적는 편이 정확하다.

| Gap                             | 실제 코드 근거                                                                        | A2A 요구                                                                                         | 영향도    |
|---------------------------------|---------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------------|--------|
| Agent discovery / Agent Card 부재 | `.cmux/agents.json`, `cmux_agent/cli/commands.py::_load_config()`               | Agent Card 메타데이터: `supportedInterfaces`, `skills`, `capabilities`                               | Large  |
| Task 엔티티와 상태 머신 부재              | `cmux_agent/domain/models.py`에 `Run`, `Message`만 존재                             | `Task.id`, `status.state`, `history`, `artifacts`                                               | Large  |
| Message correlation 정보 부족       | `Message`에 `task_id`, `context_id` 없음, `build_delivery()`의 `message_id`는 `None` | `Message.messageId`, `taskId`, `contextId`                                                      | Large  |
| Part / Artifact 모델 부재           | 모든 payload가 문자열 또는 자유 dict                                                      | `Part(text/raw/url/data)`, `Artifact.parts[]`                                                   | Medium |
| 표준 메서드/transport 부재             | `type=dispatch/result` + 파일 감시 방식                                               | `SendMessage`, `GetTask`, `ListTasks`, `CancelTask`, `SubscribeToTask` over JSON-RPC/HTTP/gRPC  | Large  |
| 스트리밍/구독 부재                      | `CmuxAdapter.notify()`와 `send_text()`는 로컬 side effect일 뿐                        | SSE, push notification, task subscription                                                       | Large  |
| 다자간 peer 모델의 제한                 | prompt와 `cmd_task()`는 orchestrator→worker, worker→orchestrator 흐름을 전제           | 임의 agent가 임의 agent와 task continuity를 유지                                                         | Small  |

추가로 문서화할 가치가 있는 세부 사항도 있다.

- `ArtifactWatcher`는 `.xml` 확장자도 허용하지만 실제 구현은 `json.loads()`만 호출한다. 따라서 "다중 포맷 transport"로 보기 어렵다.
- 일부 live inbox 파일에는 `task`, `result`, `instructions`, `artifact_format`이 들어 있지만, 현재 코드에서 이 필드를 만드는 경로는 찾지 못했다.
  canonical schema는 watcher가 검증하는 최소 outbox 형식과 `build_delivery()`가 생성하는 단순 inbox 형식이다.

## 2. 마이그레이션 전략

### 2.1 원칙

- 목표는 "로컬 내부 도구에 A2A 개념을 적용한 내부 모델 정렬"이지, 공개 서비스나 범용 프로토콜 서버를 만드는 것이 아니다.
- 기존 worker/orchestrator prompt와 `dispatch`/`result` artifact 포맷은 깨지지 않아야 한다.
- 같은 노트북 안의 여러 세션, 또는 사내 VPN/사설망 안의 다른 개발자 런타임과 연결할 필요가 생기기 전까지는 파일 transport와 cmux UX를 유지하는 편이 맞다.
- `SendMessage`/`GetTask` 같은 표면을 실제로 노출하기 전까지는 "A2A-shaped internal model"로 보는 것이 정확하다.

### 2.2 Phase 1: 내부 모델 정렬, 파일 transport 유지

이 단계는 가장 현실적이다. 현재 브로커 경계에서 legacy artifact를 A2A-like 내부 객체로 정규화하면 된다.

**변경 대상 파일/클래스**

| 파일                                                                                                                          | 변경 포인트                                                                  | 이유                                                            |
|-----------------------------------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------|---------------------------------------------------------------|
| `cmux_agent/domain/models.py`                                                                                               | `TaskState`, `Task`, `Part`, `Artifact` 추가 또는 `Message` 확장              | 현재 `Run`과 `Message`만으로는 A2A `Task`와 `Part`를 표현할 수 없다.         |
| `cmux_agent/infrastructure/storage.py`                                                                                      | `tasks`, `artifacts` 테이블 추가 또는 `messages` 확장                            | `task_id`, `context_id`, `state`, artifact 목록을 저장해야 한다.       |
| `cmux_agent/application/broker.py`                                                                                          | dispatch 수신 시 task 생성/진행 상태 기록, result 수신 시 artifact 추가 및 완료 처리         | 지금은 message 레코드만 남긴다.                                         |
| `cmux_agent/application/prompting.py`                                                                                       | `message_id`, `task_id`, `context_id`, `parts`를 담는 additive delivery 생성 | inbox JSON이 최소한 A2A correlation 정보를 잃지 않아야 한다.                |
| `cmux_agent/application/watcher.py`                                                                                         | legacy 4-field artifact와 확장 artifact를 모두 허용                             | 기존 worker protocol 호환을 유지해야 한다.                               |
| `cmux_agent/cli/commands.py`                                                                                                | string config와 object config를 모두 읽는 loader로 확장                          | Agent Card 메타데이터를 수용하되 기존 `.cmux/agents.json`을 깨지 않기 위해 필요하다. |
| `tests/test_models.py`, `tests/test_storage.py`, `tests/test_broker.py`, `tests/test_prompting.py`, `tests/test_watcher.py` | task/part/correlation/backward compatibility 테스트 추가                     | Phase 1의 핵심 리스크는 기존 artifact 호환성 회귀다.                         |

**호환성 영향 범위**

| 변경 항목               | 깨질 수 있는 지점                  | 권장 호환 전략                                                             |
|---------------------|-----------------------------|----------------------------------------------------------------------|
| `agents.json` 구조 확장 | 기존 저장소의 string-only config  | string이면 기존 provider로 해석하고, object면 새 필드를 읽는 이중 파서 유지                |
| inbox delivery 확장   | 기존 prompt가 `message` 필드만 참조 | `message`는 유지하고, 그 옆에 `task_id`, `context_id`, `parts`를 추가           |
| outbox artifact 확장  | 기존 worker가 최소 4개 필드만 생성     | watcher가 legacy schema를 받아 내부에서 task/message로 래핑                     |
| DB schema 변경        | 기존 run/state 조회             | migration 또는 additive table 생성으로 기존 `runs`, `agents`, `messages`는 유지 |

**Before / After 예시 1: `Message`만 있던 모델을 Task 중심으로 확장**

Before:

```python
@dataclass
class Message:
    message_id: str = field(default_factory=_uuid)
    run_id: str = ""
    sender: str = ""
    recipient: str = ""
    type: MessageType = MessageType.DISPATCH
    status: MessageStatus = MessageStatus.PENDING
    payload: str = ""
```

After (권장 방향):

```python
class TaskState(str, enum.Enum):
    SUBMITTED = "SUBMITTED"
    WORKING = "WORKING"
    COMPLETED = "COMPLETED"
    FAILED = "FAILED"


@dataclass
class Part:
    kind: str
    text: str | None = None
    data: dict | None = None
    media_type: str = "text/plain"


@dataclass
class Task:
    task_id: str = field(default_factory=_uuid)
    context_id: str = field(default_factory=_uuid)
    run_id: str = ""
    state: TaskState = TaskState.SUBMITTED
    history_message_ids: list[str] = field(default_factory=list)
```

**Before / After 예시 2: correlation 정보를 잃지 않는 delivery 생성**

Before (`cmux_agent/application/prompting.py` 현재 구현):

```python
return {
    "message_id": None,
    "from": sender,
    "type": msg_type.value.lower(),
    "message": payload.get("message", ""),
    "context": payload.get("context", {}),
    "created_at": now,
}
```

After (권장 방향, legacy 호환 유지):

```python
return {
    "message_id": message_id,
    "from": sender,
    "type": msg_type.value.lower(),
    "message": payload.get("message", ""),  # legacy consumer 호환
    "context": payload.get("context", {}),  # legacy consumer 호환
    "task_id": task_id,
    "context_id": context_id,
    "parts": [
        {"kind": "text", "text": payload.get("message", ""), "media_type": "text/plain"}
    ],
    "created_at": now,
}
```

**Before / After 예시 3: `agents.json`을 string/object 겸용으로 읽기**

Before:

```python
with path.open(encoding="utf-8") as f:
    return {**DEFAULT_CONFIG, **json.load(f)}
```

After (권장 방향):

```python
raw = json.load(f)
normalized = {}
for name, value in raw.items():
    if isinstance(value, str):
        normalized[name] = {"provider": value}
    else:
        normalized[name] = value
return {**DEFAULT_CONFIG_OBJECTS, **normalized}
```

이 단계가 끝나면 파일 transport는 유지되지만, 내부적으로는 "dispatch → task 생성", "result → task 완료 + artifact 생성"이라는 A2A 대응 관계를 만들 수 있다.

### 2.3 Phase 2: 선택적 내부 transport adapter 추가

이 단계는 "여러 로컬 런타임끼리 프로그래밍 방식으로 연결해야 하는가?"가 명확해졌을 때만 진행하면 된다. 대상은 공개 인터넷이 아니라 같은 머신의 다른 세션, 또는 사내망 안의 다른 agent runtime이다.

- 파일 기반 watcher/broker는 그대로 두고, 그 앞에 loopback HTTP, Unix domain socket, 또는 사내망 전용 JSON-RPC adapter를 추가한다.
- adapter는 `SendMessage`, `GetTask`, `ListTasks` 같은 내부용 메서드 표면만 제공하고, 실제 상태는 기존 `StateStore`와 `MessageBroker`를 재사용한다.
- `CmuxAdapter.send_text()` 기반 주입은 transport가 아니라 로컬 UX adapter로 유지한다.
- `.cmux/agents.json` 확장 포맷에 내부 endpoint 또는 socket 경로 같은 선택 필드를 넣을 수는 있지만, 현재처럼 string-only config도 계속 허용해야 한다.

문서 초안의 "FastAPI 등 웹 프레임워크 의존성 추가 필요 (이미 의존성에 포함)" 문장은 현재 코드 기준으로 틀리다. `pyproject.toml`의 runtime dependency는 `watchdog` 하나뿐이므로, 내부 transport adapter를 붙이더라도 웹 프레임워크나 RPC 런타임은 새로 추가해야 한다.

### 2.4 선택 과제: 내부 다중 세션 협업 기능

로컬/사내 도구라는 전제를 유지하면 Phase 3를 독립 단계로 크게 둘 필요는 없다. 아래 항목은 실제 수요가 생길 때만 소규모 후속 작업으로 추가하면 된다.

- `Task` 진행률 이벤트를 `events.jsonl`과 별도로 스트리밍하거나 watch할 필요가 있는지 검토
- `CancelTask` 같은 취소 semantics가 실제 운영 플로우에 필요한지 검증
- 여러 개발자 세션이 동시에 같은 run/task를 건드릴 경우의 충돌 정책을 `StateStore`와 broker 수준에서 정리

## 3. 트레이드오프

### 3.1 현재 파일 기반 transport가 실제로 잘하는 것

현재 구조의 장점은 추상적인 "단순함"이 아니라, 코드로 확인되는 운영상 이점에 있다.

- **디버깅이 쉽다**: `.cmux/outbox`, `.cmux/inbox/{agent}`, `.cmux/processed`, `events.jsonl`, `control-plane.sqlite3`를 직접 보면
  전체 흐름이 재현된다.
- **원자적 inbox 쓰기**: `AgentFileSystem.write_to_inbox()`는 `.tmp` 파일에 쓴 뒤 rename한다.
- **재시작 복구가 단순하다**: `ArtifactWatcher._process_existing()`가 watcher 시작 시 기존 outbox 파일을 다시 훑는다.
- **의존성이 매우 작다**: 현재 runtime dependency는 `watchdog` 하나뿐이다.
- **cmux와의 결합이 명확하다**: `CmuxAdapter.notify()`, `send_text()`, `send_key()`로 로컬 UX를 바로 제어할 수 있다.

특히 transport 핵심 구현은 `cmux_agent/application/broker.py`(220줄), `cmux_agent/application/watcher.py`(130줄),
`cmux_agent/infrastructure/filesystem.py`(69줄), `cmux_agent/infrastructure/storage.py`(205줄)로 합계 624줄 수준이다. 로컬 브로커로서는
상당히 작은 편이다.

### 3.2 A2A 개념 도입이 주는 실질적 이익

- **내부 인터페이스 일관성**: 현재는 cmux-agent 내부 prompt/protocol을 알아야만 붙을 수 있지만, A2A식 `Task`/`Message`/`Part` 구조를 도입하면 같은 팀의 다른 런타임도 더 예측 가능한 방식으로 붙을 수 있다.
- **작업 추적 정교화**: 지금은 `Run`과 `MessageStatus`만 있어 "세션은 살아있다"와 "메시지는 전달됐다"만 추적한다. A2A task state를 도입하면 작업별 진행/실패/입력 대기 상태를
  구분할 수 있다.
- **typed payload**: 현재 `message` 문자열 하나에 모든 의미를 싣지만, A2A `Part`/`Artifact`를 쓰면 text/data/url/raw를 분리할 수 있다.
- **브로커 경계의 명확화**: `MessageBroker._route_message()`와 `PromptBuilder.build_delivery()`를 중심으로 내부 표준이 생기면, cmux UX와 durable transport를 분리해서 다루기 쉬워진다.

### 3.3 A2A 개념 도입 비용의 정량적 평가

아래 수치는 현재 파일 크기와 책임 분포를 기준으로 한 대략적 추정치다.

| 단계 | 추가 코드/의존성 추정 | 근거 |
|------|----------------------|------|
| Phase 1 | 약 150~250 LOC, 새 runtime dependency 0개 | 기존 `models.py`, `storage.py`, `broker.py`, `prompting.py`, 테스트만 additive로 확장하면 충분하다. |
| Phase 2 | 약 200~400 LOC, runtime dependency +1~2개 | loopback HTTP/JSON-RPC 또는 Unix socket adapter, task 조회 API, request validation이 새로 필요하다. 현재 `pyproject.toml`에는 웹/RPC 프레임워크가 없다. |
| 선택 과제 | 약 80~180 LOC, dependency 추가는 선택적 | 취소/진행률 이벤트/다중 세션 조정처럼 실제 내부 운영 수요가 생긴 경우에만 얹는 작은 후속 작업이다. |

증가하는 복잡도는 단순 LOC보다 "새 책임의 수"에서 더 크게 체감된다.

- 현재는 파일 감시 + SQLite + cmux CLI 세 가지 책임만 있다.
- Phase 2부터는 여기에 local RPC lifecycle, method dispatch, schema serialization이 추가된다.
- 선택 과제까지 가면 취소 semantics, 진행률 전달, 다중 세션 충돌 처리까지 관리해야 한다.

### 3.4 최종 권고

- **권장: Phase 1만 먼저 진행**. 내부 모델을 A2A에 맞춰 정렬하되 파일 transport와 cmux UX는 그대로 둔다.
- **조건부 진행: Phase 2는 내부 다중 런타임 연동 요구가 생길 때만**. 아직 현재 코드에도 task/entity/correlation 정리가 안 되어 있어서, 이 상태에서 adapter부터 붙이면 transport만 복잡해지고 내부 모델은 더 뒤엉킬 가능성이 높다.
- **지금 하지 않을 것**: 인증 체계, 공개 인터페이스 노출, 외부 연합형 연결은 이 프로젝트의 현재 목적과 맞지 않으므로 범위에서 제외한다.
- **명확한 선 긋기**: 내부 adapter가 실제로 `SendMessage`/`GetTask` 수준의 호출을 제공하기 전까지는 "A2A-friendly internal model"이지 프로토콜 endpoint는 아니다.
